推奨アーキテクチャの考え方
コアコンセプト：C#アプリをMCPサーバー化する
最もシンプルで効果的な構成は、C#アプリ内にMCPサーバーを組み込むことです。AIボットがMCPクライアントとしてツールを呼び出すと、C#アプリが受信して既存のIPC経由でC++本体を操作します。

実装の3レイヤー
レイヤー1：WebアプリAIボット側
Claudeなどの AIを呼び出す Webアプリで、ユーザーが「この検査レシピを調整して」などと自然言語で指示します。AIはMCPツールとして定義された操作（set_threshold, run_inspection, get_result など）を呼び出します。

レイヤー2：MCPサーバー（ブリッジ）
C#アプリ内に実装するか、別プロセスとして立てるかの2択があります。

オプションA：C#アプリ内にMCPサーバーを組み込む（推奨）

C#アプリが直接MCPサーバーを起動（Named PipeかHTTPで待ち受け）
MCPツール呼び出し → 既存のIPCでC++本体に転送
C#のUIはそのまま確認インターフェースとして機能
追加プロセス不要でシンプル
オプションB：中間ブリッジサーバーを別途立てる

Node.js や Python のMCPサーバーを別プロセスで起動
そこからC#アプリをHTTP APIやCOM経由で操作
C#アプリの改修を最小限にしたいときに有効
レイヤー3：C++本体
変更不要です。C#アプリからの既存IPCをそのまま使います。

csharp
using ModelContextProtocol.Server;
using System.ComponentModel;

[McpServerToolType]
public class InspectionTools
{
    private readonly IpcClient _ipc;
    private readonly ResultCache _cache;

    public InspectionTools(IpcClient ipc, ResultCache cache)
    {
        _ipc = ipc;
        _cache = cache;
    }

    [McpServerTool, Description("画像検査を実行する")]
    public async Task<string> RunInspection()
    {
        var result = await _ipc.SendAsync("RUN_INSPECTION");
        _cache.SetLatest(result);
        return result.ToJson();
    }

    [McpServerTool, Description("検査閾値を変更する")]
    public async Task<string> SetThreshold(
        [Description("閾値 0.0〜1.0")] double value)
    {
        await _ipc.SendAsync($"SET_THRESHOLD:{value}");
        return $"閾値を {value} に設定しました";
    }

    [McpServerTool, Description("最新の検査結果を取得する")]
    public Task<string> GetResult()
    {
        var r = _cache.GetLatest();
        return Task.FromResult(r?.ToJson() ?? "結果なし");
    }

    [McpServerTool, Description("レシピを読み込む")]
    public async Task<string> LoadRecipe(
        [Description("レシピID")] string recipeId)
    {
        await _ipc.SendAsync($"LOAD_RECIPE:{recipeId}");
        return $"レシピ {recipeId} を読み込みました";
    }
}

public class InspectionViewModel : INotifyPropertyChanged
{
    private readonly ResultCache _cache;
    private InspectionResult? _latestResult;

    public InspectionResult? LatestResult

        set { _latestResult = value; OnPropertyChanged(); }

WCFクライアントラッパー（WcfInspectionClient.cs）
csharp
public class WcfInspectionClient : IDisposable
{
    private readonly ChannelFactory<IInspectionService> _factory;
    private IInspectionService? _channel;

    public WcfInspectionClient(string endpointAddress)
    {
        // C++側のバインディングに合わせて選択
        // ── 同一マシン内通信なら NetNamedPipeBinding（高速・推奨）
        // ── 別マシンなら NetTcpBinding
        var binding = new NetNamedPipeBinding
        {
            ReceiveTimeout    = TimeSpan.FromSeconds(30),
            SendTimeout       = TimeSpan.FromSeconds(30),
            MaxReceivedMessageSize = 10 * 1024 * 1024  // 10MB（画像パスなど）
        };

        var endpoint = new EndpointAddress(endpointAddress);
        _factory = new ChannelFactory<IInspectionService>(binding, endpoint);
    }

    private IInspectionService Channel
    {
        get
        {
            // チャネルが壊れていたら再生成
            if (_channel == null ||
                ((ICommunicationObject)_channel).State == CommunicationState.Faulted)
            {
                _channel = _factory.CreateChannel();
                ((ICommunicationObject)_channel).Open();
            }
            return _channel;
        }
    }

    public Task<InspectionResult> RunInspectionAsync() =>
        Task.Run(() => Channel.RunInspection());

    public Task SetThresholdAsync(double value) =>
        Task.Run(() => Channel.SetThreshold(value));

    public Task LoadRecipeAsync(string recipeId) =>
        Task.Run(() => Channel.LoadRecipe(recipeId));

    public Task<InspectionResult> GetLatestResultAsync() =>
        Task.Run(() => Channel.GetLatestResult());

    public Task SaveRecipeAsync(string recipeId) =>
        Task.Run(() => Channel.SaveRecipe(recipeId));

    public void Dispose()
    {
        try { ((ICommunicationObject?)_channel)?.Close(); } catch { }
        try { _factory.Close(); } catch { }
    }
}

ResultCache ― スレッドセーフな結果受け渡し
csharp
public class ResultCache
{
    private InspectionResult? _latest;
    private readonly object _lock = new();

    // MCPツールからUIへのイベント通知
    public event Action<InspectionResult>? ResultUpdated;

    public void SetLatest(InspectionResult result)
    {
        lock (_lock) { _latest = result; }
        ResultUpdated?.Invoke(result);   // ViewModelに通知
    }

    public InspectionResult? GetLatest()
    {
        lock (_lock) { return _latest; }
    }
}

注意点まとめ
WCFのスレッド問題について。Task.Run() でWCF呼び出しをバックグラウンドに逃がしていますが、WCFのチャネルはスレッドセーフではないため、WcfInspectionClient 内部でチャネル生成をロックするか、呼び出しごとに新しいチャネルを作る方式が安全です。高頻度で検査を回す場合はチャネルプールの検討も視野に入れてください。

エンドポイントURLの確認は、C++側のApp.configかweb.configに記載されているバインディング設定から持ってきます。net.pipe:// ならNetNamedPipeBinding、net.tcp:// なら NetTcpBinding を使います。

IPCの具体的なエンドポイントURLや、C++側がどちらのバインディングを使っているかを教えていただければ、接続確認コードも含めてさらに詳しくお出しできます。

[McpServerToolType]という記述はどういう役割を果たしますか

6月14日
[McpServerToolType] はクラスレベルの属性（Attribute）で、「このクラスの中にMCPツールが入っています」とフレームワークに教えるマーカーです。

登録の仕組み
csharp
// App.xaml.cs の登録時
services.AddMcpServer()
    .WithToolsFromAssembly(typeof(InspectionTools).Assembly);
WithToolsFromAssembly() が呼ばれると、フレームワークがアセンブリ内のクラスを全スキャンして、[McpServerToolType] が付いているクラスだけを探します。見つかったクラスの中の [McpServerTool] メソッドをMCPツールとして登録する、という2段階の仕組みです。

アセンブリ全体をスキャン
    └─ [McpServerToolType] が付いたクラスを発見
            └─ その中の [McpServerTool] メソッドを登録
                    ├─ RunInspection
                    ├─ SetThreshold
                    └─ ...
付けない場合との違い
csharp
// [McpServerToolType] あり → 自動スキャンで発見される
[McpServerToolType]
public class InspectionTools { ... }

// [McpServerToolType] なし → スキャンで無視される
public class InspectionTools { ... }
WithToolsFromAssembly() を使う代わりに、クラスを個別に明示登録する方法もあります。その場合は [McpServerToolType] は不要です。

csharp
// 個別登録なら属性なしでも動く
services.AddMcpServer()
    .WithTools<InspectionTools>();   // 明示的に指定
使い分けの目安
方法	[McpServerToolType]	向いている場面
WithToolsFromAssembly()	必要	ツールクラスが複数あり、増える予定がある
WithTools<T>()	不要	ツールクラスが1〜2個で固定
今回のケースはツールクラスが InspectionTools 1つなので、WithTools<InspectionTools>() の個別登録の方がシンプルで分かりやすいかもしれません。

const response = await fetch("http://localhost:5050/mcp", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
        jsonrpc: "2.0",
        method: "tools/call",
        params: {
            name: "RunInspection",
            arguments: {}
        },
        id: 1
    })
});
ただしこの方法にはひとつ大きな注意があります。

重要な注意：ネットワーク到達性
C#アプリ（MCPサーバー）は工場内のWindowsマシン上で動いているはずです。Webアプリがクラウドや別ネットワークにある場合、直接はアクセスできません。

Webアプリの場所	C#アプリへの到達	対策
同一マシン上	✅ localhost で直接OK	そのまま使える
同一LAN内	✅ IPアドレス指定でOK	ファイアウォール開放が必要
クラウド・外部	❌ 直接不可	リバースプロキシ・VPN・トンネルが必要
外部からアクセスしたい場合は、ngrok のようなトンネルツールで一時的に公開するか、社内VPN経由でアクセスする構成が現実的です。

どちらのパターンにしますか？
「ユーザーが自然言語で指示→AIが判断して検査操作」ならパターンA
「Webアプリのボタンで特定の検査操作を直接実行」ならパターンB

@app.post("/chat")

    response = client.beta.messages.create(

    reply = next(
        (block.text for block in response.content if block.type == "text"),
        "応答を取得できませんでした"
    )

    return {
        "reply": reply,
        "usage": {
            "input_tokens":  response.usage.input_tokens,
            "output_tokens": response.usage.output_tokens,
        }
    }

const history = [];

async function send() {
  const input = document.getElementById("input");
  const btn   = document.getElementById("btn");
  const msg   = input.value.trim();
  if (!msg) return;

  addMessage("user", msg);
  input.value = "";
  btn.disabled = true;
  addMessage("ai", "考え中...");

  try {
    const res = await fetch("/chat", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({
        message: msg,
        conversation_history: history
      })
    });
    const data = await res.json();

    // 最後の「考え中...」を置き換え
    replaceLastAiMessage(data.reply);

    // 会話履歴を更新
    history.push({ role: "user",      content: msg         });
    history.push({ role: "assistant", content: data.reply  });

  } catch (e) {
    replaceLastAiMessage("エラーが発生しました: " + e.message);
  } finally {
    btn.disabled = false;
    input.focus();
  }
}

function addMessage(role, text) {
  const log = document.getElementById("log");
  const div = document.createElement("div");
  div.className = role === "user" ? "msg-user" : "msg-ai";
  div.innerHTML = `<span>${text}</span>`;
  log.appendChild(div);
  log.scrollTop = log.scrollHeight;
}

function replaceLastAiMessage(text) {
  const log  = document.getElementById("log");
  const msgs = log.querySelectorAll(".msg-ai");
  const last = msgs[msgs.length - 1];
  if (last) last.querySelector("span").textContent = text;
}

// Enterキーで送信
document.getElementById("input").addEventListener("keydown", e => {
  if (e.key === "Enter") send();
});

2. /api/chat エンドポイント（中核部分）
ブラウザから送られてきたメッセージを受け取り、Claudeを呼び出して返す処理です。

ステップ①：会話履歴の組み立て

messages = req.conversation_history + [
    {"role": "user", "content": req.message}
]
ブラウザ側がこれまでの会話履歴をすべて持って送ってきます。Claudeはメモリを持たないため、毎回全履歴を渡すことで文脈を維持しています。

ステップ②：Claude API呼び出し（最重要）
python
response = client.beta.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system=SYSTEM_PROMPT,     # ← AIへの役割指示
    messages=messages,        # ← 会話履歴
    mcp_servers=[{
        "type": "url",
        "url": MCP_SERVER_URL,          # ← C#アプリのMCPサーバー
        "name": "inspection-server",
    }],
    betas=["mcp-client-2025-04-04"],    # ← MCP機能を有効化
)
ここが最も重要な部分です。mcp_servers にC#アプリのURLを渡すと、Claude APIが自動的にMCPサーバーへ接続してツール一覧を取得し、ユーザーの指示に応じて適切なツールを選んで呼び出します。Webサーバーのコードは「どのツールを呼ぶか」を一切判断しません。 その判断はすべてClaude側が行います。

内部でClaudeは次の処理を自動的に行っています。

① C#のMCPサーバーに tools/list を送って利用可能ツールを取得
② ユーザーのメッセージを読んで、どのツールを呼ぶか判断
③ C#のMCPサーバーに tools/call を送ってツールを実行
④ ツールの実行結果を受け取って自然言語の応答を生成
ステップ③：応答の抽出
python
# テキスト応答を取り出す
reply = next(
    (block.text for block in response.content if block.type == "text"),
    "応答を取得できませんでした"
)

# 呼ばれたツール名を取り出す（画面のバッジ表示用）
tool_calls = [
    block.name
    for block in response.content
    if block.type == "tool_use"
]
response.content は複数のブロックの配列になっています。ツール呼び出しがあった場合、たとえばこういう構造になっています。

json
[
  { "type": "tool_use",  "name": "RunInspection", "input": {} },
  { "type": "tool_use",  "name": "SetThreshold",  "input": {"value": 0.9} },
  { "type": "text",      "text": "検査を実行しました。結果は合格（スコア: 0.94）です。" }
]
type == "text" のブロックだけをユーザーへの返答として取り出し、type == "tool_use" のブロックからはツール名だけを取り出して画面のバッジ表示に使っています。

from sentence_transformers import SentenceTransformer
import PyPDF2, pathlib, textwrap

DOCS_DIR  = "./docs"        # マニュアルPDFを置くフォルダ
CHUNK_SIZE = 600            # 1チャンクの文字数

model = SentenceTransformer("paraphrase-multilingual-MiniLM-L12-v2")
client = chromadb.PersistentClient(path="./chroma_db")
col = client.get_or_create_collection("manuals")

def split_text(text: str, size: int) -> list[str]:
    return textwrap.wrap(text, size, break_long_words=False)

for pdf_path in pathlib.Path(DOCS_DIR).glob("*.pdf"):
    reader = PyPDF2.PdfReader(str(pdf_path))
    full_text = "\n".join(page.extract_text() or "" for page in reader.pages)

    chunks = split_text(full_text, CHUNK_SIZE)
    vectors = model.encode(chunks).tolist()

    col.add(
        documents=chunks,
        embeddings=vectors,
        ids=[f"{pdf_path.stem}_{i}" for i in range(len(chunks))],
        metadatas=[{"source": pdf_path.name} for _ in chunks],
    )
    print(f"登録完了: {pdf_path.name} ({len(chunks)} チャンク)")

def search_manual(query: str, n_results: int = 3) -> str:
    """クエリに最も近いマニュアルの箇所を返す"""
    vector = _embed_model.encode([query]).tolist()
    results = _col.query(query_embeddings=vector, n_results=n_results)

    return "\n\n".join(lines)

# ingest.py（API仕様書版）
import chromadb, re
from sentence_transformers import SentenceTransformer

model  = SentenceTransformer("paraphrase-multilingual-MiniLM-L12-v2")
client = chromadb.PersistentClient(path="./chroma_db")
col    = client.get_or_create_collection("api_spec")

def split_by_function(md_text: str) -> list[dict]:
    """## 関数名 の見出しで分割する"""
    chunks = []
    sections = re.split(r'^## ', md_text, flags=re.MULTILINE)
    for section in sections:
        if not section.strip():
            continue
        lines    = section.strip().splitlines()
        func_name = lines[0].strip()
        body      = "\n".join(lines).strip()
        chunks.append({"name": func_name, "text": body})
    return chunks

with open("./docs/api_spec.md", encoding="utf-8") as f:
    md = f.read()

chunks  = split_by_function(md)
texts   = [c["text"] for c in chunks]
vectors = model.encode(texts).tolist()

col.add(
    documents=texts,
    embeddings=vectors,
    ids=[c["name"] for c in chunks],
    metadatas=[{"function": c["name"]} for c in chunks],
)
print(f"API仕様登録完了: {len(chunks)} 関数")

システムの要件によって最適解が変わります。

要件	推奨構成
自然言語で柔軟に操作したい（現在）	MCP または Function Calling
C#側をシンプルに保ちたい	Function Calling + REST API
検査完了を即座にブラウザへ反映したい	SignalR を追加

将来的に他のAIツールとも連携したい	MCP（標準プロトコルの強み）
現在のMCPを使い続けることへの評価
MCPは「AIツールの標準インターフェース」として業界で採用が広がっており、将来的にClaude以外のAIからも同じC#アプリを操作できるという拡張性があります。一方でまだ新しい技術であり、ドキュメントやトラブル事例が少ないという現実的な課題もあります。

現在の構成で最も改善効果が高いとすれば、C#側はMCPのまま維持しつつ、Webサーバー側にSignalRを加えてC#アプリからの自動通知を受け取れるようにすることです。これにより「AIが操作した結果がブラウザにリアルタイムで表示される」という体験が実現し、オペレーターの確認作業がより自然になります。

// Services/ManualIndexService.cs（新規追加）
public class ManualIndexService
{
    // マニュアルページのインデックス
    // 実際はDBやJSONファイルから読み込む
    private readonly List<ManualPage> _index = [
        new("キャリブレーション", "manual.pdf", 12,  "センサーキャリブレーション手順"),
        new("照明",               "manual.pdf", 24,  "照明ユニットの調整"),
        new("エラー E-042",       "manual.pdf", 87,  "エラーコード一覧と対処法"),
        new("レシピ作成",         "manual.pdf", 34,  "検査レシピの作成手順"),
    ];

    public Task<ManualPage> FindPageAsync(string keyword)
    {
        // キーワードで部分一致検索
        var page = _index.FirstOrDefault(p =>
            p.Keywords.Contains(keyword, StringComparison.OrdinalIgnoreCase) ||
            p.Title.Contains(keyword,    StringComparison.OrdinalIgnoreCase))
            ?? new ManualPage(keyword, "", 0, "") { Found = false };

        return Task.FromResult(page);
    }

    public Task<List<ManualPage>> GetTableOfContentsAsync()
        => Task.FromResult(_index);
}

public class ManualPage
{
    public string Keywords   { get; set; }
    public string FilePath   { get; set; }
    public int    PageNumber { get; set; }
    public string Title      { get; set; }
    public bool   Found      { get; set; } = true;

    public ManualPage(string keywords, string filePath, int page, string title)
    {
        Keywords = keywords; FilePath = filePath;
        PageNumber = page;   Title = title;
    }
}

# main.py
def load_prompt(*filenames: str) -> str:
    """prompts/フォルダのファイルを結合して読み込む"""
    base = os.path.join(os.path.dirname(__file__), "prompts")
    parts = []
    for name in filenames:
        path = os.path.join(base, name)
        if os.path.exists(path):
            with open(path, encoding="utf-8") as f:
                parts.append(f.read())
    return "\n\n".join(parts)

SYSTEM_PROMPT = load_prompt(
    "system_prompt.md",
    "rules_manual.md",
    "rules_error.md",
    "rules_style.md",
)

6月14日
良い視点です。整理すると3つのアプローチがあります。

問題の構造
system_promptが大きいほど、毎回のAPI呼び出しでinput_tokensが増えます。

毎回のコスト = 会話履歴 + system_prompt(ルール全部) + ユーザーメッセージ
                              ↑ここが無駄に大きい
アプローチ1：ルールを階層化して必要なときだけ渡す
常に必要なルールと、特定状況でだけ必要なルールを分けます。

python
# 常に渡す（小さく保つ）
BASE_PROMPT = load_prompt("system_prompt.md")  # 役割・基本ツール一覧のみ

# 状況に応じて追加
RULE_CHUNKS = {
    "manual":  load_prompt("rules_manual.md"),
    "error":   load_prompt("rules_error.md"),
    "night":   load_prompt("rules_night.md"),
}

def build_prompt(message: str) -> str:
    rules = [BASE_PROMPT]

    # メッセージ内容でルールを動的に追加
    if any(w in message for w in ["マニュアル", "方法", "わからない", "できない"]):
        rules.append(RULE_CHUNKS["manual"])
    if any(w in message for w in ["エラー", "異常", "アラーム"]):
        rules.append(RULE_CHUNKS["error"])

    return "\n\n".join(rules)
シンプルで即効性がありますが、キーワードマッチの精度に限界があります。

アプローチ2：Prompt Caching（最もコスト効果が高い）
Anthropicが提供するプロンプトキャッシュ機能を使います。同じsystem_promptは初回だけ処理してキャッシュし、2回目以降はキャッシュ済みトークンとして大幅割引で処理されます。

python
response = client.beta.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": SYSTEM_PROMPT,
            "cache_control": {"type": "ephemeral"}  # ← これだけ追加
        }
    ],
    messages=messages,
    mcp_servers=[...],
    betas=["mcp-client-2025-04-04", "prompt-caching-2024-07-31"],
)
キャッシュされるとinput_tokensのコストが約90%削減されます。system_promptが大きいほど効果が高く、今回のケースに最も適しています。キャッシュの有効期間は5分で、5分以内に次のリクエストがあれば自動延長されます。

コスト比較のイメージです。

キャッシュなし: system_prompt(2000トークン) × リクエスト数 = 全額
キャッシュあり: 初回だけ全額、2回目以降は約10%のコスト
アプローチ3：ルールをMCPツールとして持たせる（根本的な解決）
ルール自体をAIが「必要なときに取得するツール」として定義します。system_promptにはルールの存在だけを伝え、詳細は必要時にのみ取得します。

python
# system_promptは最小限
SYSTEM_PROMPT = """
あなたは画像検査システムのアシスタントです。
対応方法が不明な場合は GetOperationRule ツールでルールを確認してください。
"""
csharp
// C#側にルール取得ツールを追加
[McpServerTool]
[Description("対応方法が不明なときや、特定状況でのルールを確認するときに呼ぶ")]
public Task<string> GetOperationRule(
    [Description("確認したいルールの種類: manual / error / style / night")]
    string ruleType)
{
    var path = Path.Combine(_rulesDir, $"rules_{ruleType}.md");
    if (!File.Exists(path))
        return Task.FromResult($"ルール '{ruleType}' が見つかりません");

    return Task.FromResult(File.ReadAllText(path));
}
この方法ではルールはAIが判断して必要なときだけ取得するため、不要なルールのトークンコストはゼロになります。

どれを選ぶか
アプローチ	効果	実装コスト	向いている場面
ルール階層化	中	低	ルール数が少ない
Prompt Caching	高	低（1行追加）	同じルールを繰り返し使う
ルールをツール化	最高	中	ルール数が多く増え続ける
現状ではPrompt Cachingが最もコスパが良いです。コード1行の追加でsystem_promptのコストが約90%削減され、ルール管理のファイル外出しとも完全に併用できます。
