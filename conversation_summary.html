設計・実装記録 — C# WPF + C++ + Claude API + MCP

画像検査AIボットシステム 設計・実装記録

概要

C#（WPF）製の画像検査編集ソフトと、C++製の画像検査システム本体（WCF通信）を、 WebアプリのAIボットで制御するシステムの設計・実装記録。

1. システム要件の整理

既存構成

  • C#アプリ（WPF）：画像検査編集ソフト（確認UI）
  • C++アプリ：画像検査システム本体
  • IPC：WCF（Windows Communication Foundation）

追加要件

  • WebアプリのAIボットから画像検査システムを制御する
  • AIボットで指示・編集した結果はC#アプリで確認できること
  • AIがツールを選択すること（必須）
  • 将来的にC#アプリ自体もAIで制御したい

2. アーキテクチャ選定

図1：システム全体アーキテクチャ

採用構成：C#アプリ内にMCPサーバーを組み込む

ブラウザ（チャットUI）
  ↓ HTTP
Webサーバー（Python / FastAPI）
  ↓ Claude API + mcp_servers パラメータ
Claude API
  ↓ MCPプロトコル（HTTP）
C# WPFアプリ（MCPサーバー内蔵）
  ↓ WCF
C++ 画像検査エンジン

他の構成との比較検討

構成 特徴 採用可否
MCP（採用） AIが自動でツール選択・C#UI制御も可能 ✅
Function Calling C#をREST APIにするだけでよい △ C#UI制御に追加層が必要
LangChain / LlamaIndex RAGとの統合が豊富 △ 学習コスト高
ルールベース 高速・低コスト △ 自然言語の柔軟な解釈が不可

MCPを選んだ理由

  • AIがどのツールを呼ぶか自動判断できる（必須要件）
  • C#アプリのWPF UIを Dispatcher.Invoke で直接制御できる
  • C#側はMCPサーバーを組み込むだけで、追加の通信レイヤーが不要
  • 将来的に他のAIツールからも接続できる標準プロトコル

6. ツールが見つからない場合の動作設計

AIが対応ツールを持たない操作を要求された場合の動作をsystem_promptで制御。

ユーザー:「センサーのキャリブレーションをして」
  ↓ AIがツール一覧を確認 → 該当ツールなし
  ↓ GetManualTableOfContents で目次を取得
  ↓ 関連ページを特定してユーザーに提案

AI:「キャリブレーションについてはマニュアルの12ページに手順が記載されています。
    ページを開きますか？」

ユーザー:「開いて」
  ↓ OpenManualPage("キャリブレーション") を呼び出し

AI:「マニュアル『センサーキャリブレーション手順』(12ページ)を開きました。」
   → C#アプリのマニュアルビューワーに表示される

7. RAGの設計

図3：RAG構成

文書種別と対応方式

文書 方式 理由
API仕様書（小〜中） system_promptに全文埋め込み 関数間の依存関係も把握できる
API仕様書（大） 関数単位でRAG 関数1つ＝チャンク1つで分割
操作マニュアル 文字数単位でRAG 長大なため関連箇所のみ抽出

RAGのチャンク戦略

# マニュアル：文字数で均等分割
split_text(text, chunk_size=600)

# API仕様書：## 関数名 の見出しで分割
split_by_function(md_text)  # 関数1つ = チャンク1つ

検索精度向上のテクニック

検索用テキストと表示用テキストを分けることで、 「閾値を変える」→SetThreshold のようなキーワード不一致を防ぐ。

search_text  = f"{func_name} {alias_words} {description}"  # 幅広く
display_text = full_spec_text                               # 詳細

8. system_prompt管理の設計

課題

system_promptに直接ルールを書くと、ルール追加のたびにPythonファイルの編集とデプロイが必要になる。

解決策：promptをファイルで外出し管理

def load_prompt(*filenames: str) -> str:
    base = os.path.join(os.path.dirname(__file__), "prompts")
    parts = []
    for name in filenames:
        path = os.path.join(base, name)
        if os.path.exists(path):
            with open(path, encoding="utf-8") as f:
                parts.append(f.read())
    return "\n\n".join(parts)

SYSTEM_PROMPT = load_prompt(
    "system_prompt.md",
    "rules_manual.md",
    "rules_error.md",
    "rules_style.md",
)

ルールを追加するときはファイルを1つ追加するだけ。Pythonコードは触らない。

9. トークンコスト削減策

図4：Prompt Cachingによるコスト削減

課題

毎リクエストでsystem_prompt（ルール全体）を送るとinput_tokensコストが増大する。

解決策の比較

アプローチ 効果 実装コスト 向いている場面
ルール階層化（動的追加） 中 低 ルール数が少ない
Prompt Caching（推奨） 高（約90%削減） 低（1行追加） 同じルールを繰り返す
ルールをMCPツール化 最高 中 ルール数が多く増え続ける

Prompt Cachingの実装（コード1行の追加）

response = client.beta.messages.create(
    model="claude-sonnet-4-6",
    system=[
        {
            "type": "text",
            "text": SYSTEM_PROMPT,
            "cache_control": {"type": "ephemeral"}  # ← これだけ追加
        }
    ],
    messages=messages,
    mcp_servers=[...],
    betas=["mcp-client-2025-04-04", "prompt-caching-2024-07-31"],
)

キャッシュは5分間有効。5分以内に次のリクエストがあれば自動延長。

推奨ロードマップ

現在   → Prompt Caching を有効化（即日対応可・効果大）
将来   → ルール数が増えてきたらMCPツール化を組み合わせる

10. 起動手順

# 1. C++エンジン（WCFサービス）を起動

# 2. C# WPFアプリをビルド・起動
cd CSharpMcpServer
dotnet run
# → WPFウィンドウに「MCPサーバー待ち受け中 (port 5050)」と表示されればOK

# 3. Webサーバーを起動
cd WebServer
cp .env.example .env   # ANTHROPIC_API_KEY を記入
pip install -r requirements.txt
python main.py

# 4. ブラウザで http://localhost:8000 にアクセス

動作確認（MCPサーバーへの疎通確認）

curl -X POST http://localhost:5050/mcp \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"tools/list","params":{},"id":1}'
